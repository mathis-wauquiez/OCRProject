Scope

This report documents the character extraction methodology used in the OCR pipeline for historical Chinese books. The pipeline takes scanned page images as input and produces per-character segmentation masks, bounding boxes, and downstream OCR predictions.

The document covers:
- The complete extraction pipeline (CRAFT detection ‚Üí watershed segmentation ‚Üí binarization ‚Üí component assignment ‚Üí filtering)
- A known failure mode for composite characters with vertically-stacked radicals
- A proposed improvement: dual-threshold watershed with optional link score enhancement
- Configuration reference and alternative approaches considered

2. Current Extraction Methodology

Click to expand
‚ñ∂

Overview

The extraction pipeline has two main stages: (1) character detection and segmentation (GPU), and (2) patch processing and OCR recognition (CPU + GPU). This report focuses on stage 1, which is where the character boundaries are determined.

Pipeline Architecture

**Stage 1 ‚Äî Character Detection Pipeline**

```
PIL Image
 ‚Üí CRAFT neural network inference
     ‚Üí score_text (H/2, W/2)   [character region confidence]
     ‚Üí score_link (H/2, W/2)   [affinity between regions]
 ‚Üí Watershed segmentation on score_text
     ‚Üí CRAFT components (one blob per detected character)
 ‚Üí Filter CRAFT components (area, aspect ratio)
 ‚Üí Merge nearby CRAFT components (centroid distance < min_dist)
 ‚Üí Binarize original image (Otsu threshold)
     ‚Üí Image connected components (ink blobs)
 ‚Üí Filter image components (remove lines)
 ‚Üí Compute similarity: image CCs ‚Üî CRAFT components
 ‚Üí Assign each image CC to nearest CRAFT component
 ‚Üí Filter by contour proximity, size, aspect ratio, fill area
 ‚Üí Final character components
```

2.1 CRAFT Detection

Click to expand
‚ñ∂

CRAFT Model Outputs

CRAFT (Character Region Awareness For Text detection) is a convolutional neural network that produces two score maps at half the input resolution: - **score_text**: per-pixel confidence that a pixel belongs to a character region (values in [0, 1]) - **score_link**: per-pixel affinity confidence between adjacent character sub-components (values in [0, 1]) The input image is preprocessed by resizing to fit within \(\texttt{canvas\_size} = 1280\) pixels (with \(\texttt{mag\_ratio} = 5.0\)) and normalizing with ImageNet statistics. The model checkpoint is ``craft_mlt_25k.pth`` (multi-lingual text, 25k iterations). **Important**: In the current pipeline, ``score_link`` is computed but not used. This is relevant to the proposed improvement.

2.2 Watershed Segmentation

Click to expand
‚ñ∂

Watershed Algorithm

CRAFT components are extracted from ``score_text`` using watershed segmentation (``connectedComponent.from_image_watershed()``). The algorithm: 1. **Binarize**: Create a foreground mask where \(\texttt{score\_text} > \tau_t\) (``text_threshold``, default 0.6) 2. **Find seeds**: Detect local maxima in the score map within the foreground mask (``peak_local_max``, ``min_distance`` = ``min_dist``) 3. **Watershed**: Using elevation surface \(E = -\texttt{score\_text}\), expand each seed basin within the foreground mask **Critical observation**: The same threshold \(\tau_t = 0.6\) is used for both seed detection (step 2) and basin expansion mask (step 3). This means areas with moderate CRAFT scores (0.3‚Äì0.5) are excluded from the watershed entirely ‚Äî they receive no CRAFT label.

2.3 CRAFT Component Post-Processing

Click to expand
‚ñ∂

Filtering and Merging

After watershed, CRAFT components are refined in two steps:

**Filtering** ‚Äî Remove spurious detections:
- Components with area < ``min_area`` (18 px)
- Components with aspect ratio outside [``min_aspect_ratio``, ``max_aspect_ratio``] = [0.2, 5.0]

**Merging** ‚Äî Combine nearby components:
- Compute pairwise centroid distances in CRAFT space
- Build a graph where edges connect components with distance < ``min_dist`` (8.0 px in CRAFT space ‚âà 16 px in image space)
- Find connected components ‚Üí merge groups
- All labels in a group are mapped to a single representative

2.4 Image Binarization & Connected Components

Click to expand
‚ñ∂

Image Processing

In parallel with CRAFT analysis, the original image is:

1. **Binarized** using Otsu's method (foreground = ink, background = paper)
2. **Connected components** extracted via ``cv2.connectedComponentsWithStats(connectivity=4)``
3. **Filtered**: elongated components with aspect ratio > 20 and major axis > 100 px are removed (these are ruling lines, not characters)

2.5 Similarity Computation & Assignment

Click to expand
‚ñ∂

Mahalanobis Distance Assignment

Each ink connected component (image CC) is assigned to the nearest CRAFT component using centroid-based distance. For each CRAFT component \(j\) with centroid \(\boldsymbol{\mu}_j\) and inertia tensor \(\Sigma_j\), and each image CC \(i\) with centroid \(\mathbf{c}_i\), the Mahalanobis distance is: $$d_M(i, j) = (\mathbf{c}_i - \boldsymbol{\mu}_j)^T \, \Sigma_j^{-1} \, (\mathbf{c}_i - \boldsymbol{\mu}_j)$$ The similarity is \(s(i, j) = -d_M(i, j)\). Each image CC is assigned to the CRAFT component with the highest similarity (lowest Mahalanobis distance). CCs with \(\max_j s(i,j) < \tau_s\) (``similarity_threshold`` = ‚àí15) are discarded as background. The inertia tensor \(\Sigma_j\) encodes the shape of the CRAFT blob: for a vertically-elongated character, \(\Sigma_j\) has a large eigenvalue in the vertical direction, making the Mahalanobis distance smaller vertically. This is an elliptical distance metric that naturally adapts to character shape.

2.6 Post-Filtering

Click to expand
‚ñ∂

Character Filtering

After assignment, merged character components are filtered:

1. **Contour proximity filter**: Remove small components that are too close (< 50 px) to the contour of large components (> 4000 px area). This removes artifacts near borders and seals.
2. **Size filter**: Remove characters with bounding box outside [30, 30] ‚Äî [150, 150] pixels
3. **Aspect ratio filter**: Remove characters with aspect ratio > 3
4. **Fill area filter**: Remove characters where filled_area / bbox_area > 0.9
5. **Minimum area filter**: Remove characters with filled area < 200 px

3. Problem: Composite Character Misassignment

Click to expand
‚ñ∂

Failure Mechanism

Characters with vertically-stacked sub-components present a systematic extraction failure. Consider the character Ëí∏:

- The top portion (body) has high CRAFT score ‚Üí becomes a CRAFT component with a watershed seed
- The bottom portion (water radical Ê∞∫) has moderate CRAFT score (~0.3‚Äì0.5) ‚Üí **falls below** ``text_threshold`` (0.6) ‚Üí **excluded from the watershed mask entirely**
- The radical receives no CRAFT label
- During Mahalanobis assignment, the radical's ink CC centroid is spatially closer to the CRAFT blob of the character below
- **Result**: the water radical is stolen by the wrong character

This affects many composite characters: Ëí∏, ÁÉù, Èªë, ÁÖÆ, ÁÜü, and any character where a bottom radical separates into a distinct ink blob positioned closer to the next character.

Root Cause: Single-Threshold Watershed

The root cause is that a **single threshold** (``text_threshold`` = 0.6) serves two conflicting purposes:

1. **Seed detection** ‚Äî needs a high threshold to avoid spurious seeds in noisy regions
2. **Basin expansion mask** ‚Äî needs a lower threshold so that basins can grow into moderate-score areas where radicals reside

By using 0.6 for both, we get precise seeds but overly restrictive basin expansion. The radical area (score 0.3‚Äì0.5) is treated as background.

4. Proposed Solution: Dual-Threshold Watershed

Click to expand
‚ñ∂

Dual-Threshold Mechanism

We split the single ``binary_threshold`` into two independent thresholds: - **Peak threshold** \(\tau_p\) (= ``text_threshold``, default 0.6): controls where watershed seeds are placed. Only strong character regions produce seeds. - **Mask threshold** \(\tau_m\) (= ``mask_threshold``, default 0.3): controls how far watershed basins can expand. Basins grow into moderate-score areas, capturing radicals. The modified watershed algorithm: 1. Seed mask: \(M_p = \{(x,y) : \texttt{score}(x,y) > \tau_p\}\) 2. Expansion mask: \(M_m = \{(x,y) : \texttt{score}(x,y) > \tau_m\}\) 3. Seeds: ``peak_local_max(score, labels=`` \(M_p\) ``)`` 4. Watershed: ``watershed(‚àíscore, markers, mask=`` \(M_m\) ``)`` Since \(\tau_m < \tau_p\), we have \(M_p \subseteq M_m\): seeds are always within the expansion mask. The key difference is that basins can now grow into the moderate-score radical area.

4.1 Why This Works

Click to expand
‚ñ∂

Score Landscape Analysis

Consider the character Ëí∏ with the character ËÄÖ directly below it. The CRAFT score landscape in the vertical direction looks approximately like: ``` Score ^ 1.0 | ‚îå‚îÄ‚îÄ‚îê ‚Üê Ëí∏ body (seed here) | ‚îÇ ‚îÇ 0.6 ‚îÄ‚îÄ|‚îÄ‚îÄ‚î§ ‚îú‚îÄ‚îÄ œÑ_p ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ seed threshold (current) | ‚îÇ ‚îÇ 0.4 | ‚îÇ ‚îî‚îÄ‚îê ‚Üê radical area (moderate score) 0.3 ‚îÄ‚îÄ|‚îÄ‚îÄ‚î§ ‚îú‚îÄ‚îÄ œÑ_m ‚îÄ‚îÄ‚îÄ‚îÄ mask threshold (new) | ‚îÇ ‚îÇ 0.1 | ‚îî‚îÄ‚îÄ‚îÄ‚îÄ‚îò | ‚Üê inter-character gap (very low score) 0.0 |‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ background | ‚îå‚îÄ‚îÄ‚îê 1.0 | ‚îÇ ‚îÇ ‚Üê ËÄÖ body (separate seed) +‚îÄ‚îÄ‚î¥‚îÄ‚îÄ‚î¥‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚Üí y ``` With \(\tau_p = 0.6\): the Ëí∏ body gets a seed. The radical area (score ~0.4) is above \(\tau_m = 0.3\) ‚Üí included in the expansion mask. The inter-character gap (score ~0.1) is below \(\tau_m\) ‚Üí remains excluded. The Ëí∏ body's basin expands downward through the radical area (following the gradient of \(-\texttt{score}\)) and stops at the inter-character gap. The radical is now part of the Ëí∏ CRAFT component, and downstream Mahalanobis assignment works correctly.

4.2 Risk Analysis

Click to expand
‚ñ∂

Potential Risks and Mitigations

**Over-expansion risk**: If \(\tau_m\) is too low, basins could bridge inter-character gaps where the score is slightly above zero. Mitigation: the inter-character gap in historical Chinese text typically has CRAFT score < 0.1, well below the recommended \(\tau_m = 0.3\). **Noise inclusion**: A lower mask threshold could include background noise. Mitigation: seeds remain at \(\tau_p = 0.6\), so no new spurious seeds are created. Only existing basins expand. **Adjacent character merging**: Only possible if the gap between adjacent characters has score > \(\tau_m\). For well-separated vertical text, this is unlikely. **Backward compatibility**: Setting ``mask_threshold: null`` in the configuration restores the original single-threshold behavior.

5. Complementary Enhancement: Link Score Combination

Click to expand
‚ñ∂

Link Score Methodology

CRAFT's ``score_link`` output detects affinity between adjacent text sub-components. The canonical CRAFT approach (from ``craft_utils.py:29``) combines text and link scores before connected component extraction: $$\texttt{score\_combined} = \text{clip}\big(\texttt{score\_text} + \mathbb{1}[\texttt{score\_link} > \tau_L],\; 0,\; 1\big)$$ where \(\tau_L\) is the link threshold. This fills in the gap between sub-components by adding 1.0 where the link score exceeds the threshold, making the region contiguous. **This is complementary to the dual-threshold approach**: - Dual threshold works when the radical has moderate CRAFT score (0.3‚Äì0.5) - Link score works when the gap has high affinity even if the radical itself has low text score **Caveat**: The CRAFT model (``craft_mlt_25k.pth``) was trained on modern multilingual text. The ``score_link`` map was designed to connect characters within words (relevant for Latin and Korean scripts). Its behavior on historical Chinese woodblock prints ‚Äî where each character is standalone ‚Äî is uncertain. The link score may not fire between stacked radicals, or it may over-fire between adjacent characters in a column. **Recommendation**: ``link_threshold`` is disabled by default (``null``). Enable it experimentally after visual inspection of ``score_link`` on your data.

7. Alternative Approaches Considered

Click to expand
‚ñ∂

Evaluated Alternatives

**A. Bounding Box Containment with Vertical Expansion** For each CRAFT component, expand its bounding box vertically by a configurable factor. Assign ink CCs by checking containment in the expanded bbox. Pro: simple and geometric. Con: axis-aligned boxes are crude; vertical expansion factor is hard to set globally. **B. CRAFT Label Map Pixel-Level Overlap** Upscale the CRAFT watershed label map to image resolution and compute pixel overlap between each ink CC and each CRAFT region. Pro: spatially precise. Con: expensive (per-pixel mapping); does not solve the problem when the radical area has no CRAFT label (the root cause). **C. Composite Similarity Score (Mahalanobis + Spatial)** Add a bbox overlap bonus to the Mahalanobis similarity: \(S = S_{\text{Maha}} + \alpha \cdot S_{\text{bbox}}\). Pro: builds on existing code. Con: still centroid-based; adds tuning complexity. **D. Vertical Territory Assignment** Define character territories using midpoints between adjacent CRAFT bounding boxes. Pro: robust for columnar text. Con: requires column detection at a pipeline stage where it's not yet available. **E. Modified Mahalanobis with Vertical Bias** Inflate the vertical component of the CRAFT inertia tensor: \(\Sigma' = \Sigma + \text{diag}(0, \sigma_v^2)\). Pro: minimal code change. Con: also extends reach in the wrong direction (upward); hard to tune. **Why dual-threshold watershed was chosen**: It addresses the root cause (restrictive watershed mask) rather than patching downstream assignment. It is model-independent, has a single interpretable parameter (\(\tau_m\)), and is fully backward-compatible.

Testing Steps

1. **Backward compatibility**: Run with ``mask_threshold: null``, ``link_threshold: null`` and verify identical output to the unmodified pipeline.

2. **Visual inspection**: Use ``visualization.create_pipeline_summary()`` to compare CRAFT components and similarity matching before and after on pages containing composite characters (Ëí∏, ÁÉù, Èªë, ÁÖÆ, ÁÜü).

3. **Score map analysis**: Visualize the CRAFT ``score_text`` heatmap for problematic characters to confirm that the radical area has scores in the 0.3‚Äì0.5 range (validating the threshold approach).

4. **Threshold sweep**: Test ``mask_threshold`` in {0.2, 0.3, 0.4, 0.5} and monitor:
  - Total CRAFT components per page
  - Total final characters per page
  - Radical misassignment rate (manual check)

5. **Over-merging check**: Verify that adjacent characters with normal spacing are NOT merged at ``mask_threshold = 0.3``.
